## 🚨 КРИТИЧЕСКОЕ ПРАВИЛО: ИСПОЛЬЗОВАНИЕ УНИВЕРСАЛЬНОГО PRINT

### ✅ ОБЯЗАТЕЛЬНОЕ ИСПОЛЬЗОВАНИЕ УНИВЕРСАЛЬНОГО PRINT:
- ✅ **ВСЕГДА** использовать стандартный `print()` (который автоматически вызывает `universal_print`)
- ✅ **ВСЕГДА** использовать параметр `level` для указания уровня сообщения
- ✅ **НИКОГДА** не использовать префиксы в тексте сообщений (`[DEBUG]`, `[ERROR]`, `[WARNING]`, `[INFO]`, `ОШИБКА:`, `ВНИМАНИЕ:`)
- ✅ **ВСЕГДА** использовать параметры вместо префиксов в тексте

### 📋 ПРАВИЛА ВЫБОРА УРОВНЯ СООБЩЕНИЯ:

#### 🔍 DEBUG (Отладочные сообщения):
- **Использовать для:** Детальная отладочная информация о работе функций, методов, переменных
- **Примеры:** Вызовы функций, значения переменных, промежуточные состояния
- **Формат:** `print(f"Текст сообщения", level='DEBUG')`
- **Пример:** `print(f"WinetricksHandler.install() вызван с component_id={component_id}", level='DEBUG')`

#### ℹ️ INFO (Информационные сообщения):
- **Использовать для:** Обычная информация о ходе выполнения программы
- **Примеры:** Начало/завершение операций, успешные действия, общая информация
- **Формат:** `print("Текст сообщения")` или `print("Текст сообщения", level='INFO')`
- **Пример:** `print("Установка компонента начата")`

#### ⚠️ WARNING (Предупреждения):
- **Использовать для:** Предупреждения о потенциальных проблемах, не критичных ошибках
- **Примеры:** Недостаточно ресурсов, пропуск операций, нестандартные ситуации
- **Формат:** `print("Текст сообщения", level='WARNING')`
- **Пример:** `print("Не все процессы остановлены, продолжаем", level='WARNING')`

#### ❌ ERROR (Ошибки):
- **Использовать для:** Критичные ошибки, сбои операций, проблемы требующие внимания
- **Примеры:** Ошибки установки, сбои проверок, критические проблемы
- **Формат:** `print("Текст сообщения", level='ERROR')`
- **Пример:** `print("Не удалось установить компонент", level='ERROR')`

### 🔧 ДОПОЛНИТЕЛЬНЫЕ ПАРАМЕТРЫ:

#### 📊 stream (Поток логирования):
- `stream='analysis'` (по умолчанию) - анализ-поток для обработанных сообщений
- `stream='raw'` - RAW-поток для необработанного вывода процессов (apt, winetricks и т.д.)
- **Пример:** `print("Вывод процесса", stream='raw')`

#### 🖥️ gui_log (Отображение в GUI):
- `gui_log=True` - отображать сообщение в GUI логе
- `gui_log=False` (по умолчанию) - не отображать в GUI
- **Пример:** `print("Сообщение для GUI", gui_log=True)`

### ❌ ЗАПРЕЩЕНО:

#### 🚫 ПРЕФИКСЫ В ТЕКСТЕ СООБЩЕНИЙ:
- ❌ **ЗАПРЕЩЕНО:** `print(f"[DEBUG] Текст сообщения")`

## 🏷️ ПРАВИЛА ВЕРСИОНИРОВАНИЯ:

### Схема версий:
- **Версия проекта**: `VMAJOR.MINOR.PATCH` (MAJOR/MINOR из `version.txt`, PATCH = номер коммита) - общая для всех файлов проекта
- **Дата релиза файла**: `YYYY.MM.DD` - индивидуальная для каждого файла

### Формат в файлах:
```bash
# Версия проекта: VMAJOR.MINOR.PATCH
# Дата релиза: YYYY.MM.DD
```

### Правила обновления:
1. **При изменении функциональности проекта** → обновляем версию проекта во ВСЕХ файлах
2. **При изменении конкретного файла** → обновляем дату релиза ТОЛЬКО в этом файле
3. **При изменении правил ASSISTANT_RULES.md** → обновляем версию проекта во ВСЕХ файлах
4. **MAJOR версия** (VVMAJOR.X.X) - кардинальные изменения архитектуры
5. **MINOR версия** (X.MINOR.X) - новые функции, улучшения
6. **PATCH версия** (X.X.PATCH) - номер коммита (сохраненных изменений)

### Примеры:
```
V2.3.76 (2025.10.25) - текущая версия проекта (76 коммитов)
├── astra_automation.py: V2.3.76 (2025.10.25) ← изменен сегодня
├── astra_install.sh: V2.3.76 (2025.10.25) ← изменен вчера
├── astra_update.sh: V2.3.76 (2025.10.25) ← изменен позавчера
├── ASSISTANT_RULES.md: V2.3.76 (2025.10.25) ← изменен сегодня
└── README.md: V2.3.76 (2025.10.25) ← обновлен сегодня
```

# ШАГ 15: Верификация коммита
echo "=== ШАГ 15: Верификация коммита ==="
if ! git log -1 --stat > /dev/null 2>&1; then
    stop_on_error "Не удалось получить информацию о последнем коммите. Коммит не был создан."
fi
git log -1 --stat

# КРИТИЧНО: Проверка создания коммита
COMMIT_HASH=$(git log -1 --format="%H")
if [ -z "$COMMIT_HASH" ]; then
    stop_on_error "Коммит не найден. Коммит не был создан."
fi
echo "✓ Коммит успешно создан: $COMMIT_HASH"

# ШАГ 16: Удаляем временный файл
echo "=== ШАГ 16: Удаление временного файла ==="
rm commit_message.txt

# ШАГ 17: Проверяем, нужна ли локальная история (изменён ли один из ключевых файлов)
# КРИТИЧНО: Используем простую проверку без сложного regex для избежания зависаний
echo "=== ШАГ 17: Проверка необходимости истории ==="
NEED_HISTORY=no
# Упрощение: объединяем три проверки в одну
if echo "$CHANGED_FILES" | grep -qE "(astra_automation\.py|astra_install\.sh|astra_update\.sh)"; then
    NEED_HISTORY=yes
fi
echo "NEED_HISTORY=$NEED_HISTORY"

# ШАГ 18: Создаем локальную историю (только если NEED_HISTORY=yes)
if [ "$NEED_HISTORY" = "yes" ]; then
  echo "=== ШАГ 18: Создание локальной истории ==="
  COMMIT_DATE=$(date +%Y_%m_%d_%H_%M)
  mkdir -p "History/$COMMIT_DATE"
  cp astra_automation.py "History/$COMMIT_DATE/" 2>/dev/null || true
  cp astra_install.sh "History/$COMMIT_DATE/" 2>/dev/null || true
  cp astra_update.sh "History/$COMMIT_DATE/" 2>/dev/null || true
  echo "=== ШАГ 19: Валидация истории ==="
  ls -la "History/$COMMIT_DATE/"
else
  echo "История не требуется (ключевые файлы не изменялись)"
fi

# ШАГ 20: Подтверждение завершения
# КРИТИЧНО: Используем только простые echo без сложных конструкций (git log, date, переменные в кавычках)
# чтобы избежать зависаний команд. НЕ использовать команды типа: echo "Дата: $(date +%Y.%m.%d)"
echo "=== ШАГ 20: Подтверждение завершения ==="
echo "Снимок (коммит) успешно создан!"
